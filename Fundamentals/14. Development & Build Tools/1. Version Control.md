# All Git global options

| **Option** | **Purpose / Description** | **Example Usage** | **Explanation** |
| --- | --- | --- | --- |
| `-v`, `--version` | Show Git version | `git --version` | Displays installed Git version and exits. |
| `-h`, `--help` | Show help | `git --help` `git commit --help` | Shows general or command-specific help. |
| `-C <path>` | Run Git as if in `<path>` | `git -C ~/projects/myrepo status` | Lets you run Git commands outside the repo folder. |
| `-c <name>=<value>` | Temporarily set config for this command | `git -c user.name="Sushil" commit -m "Test"` | Overrides config variables just for this command. |
| `--exec-path[=<path>]` | Path to Git executables | `git --exec-path=/usr/local/git-core` | Changes where Git looks for its internal programs. |
| `--html-path` | Show HTML documentation path | `git --html-path` | Prints location of Git HTML docs. |
| `--man-path` | Show man pages path | `git --man-path` | Prints location of Git man pages. |
| `--info-path` | Show info file path | `git --info-path` | Prints location of Git info files. |
| `-p`, `--paginate` | Pipe output through a pager | `git --paginate log` | Lets you scroll through output using `less` or another pager. |
| `-P`, `--no-pager` | Disable pager | `git --no-pager log` | Shows all output directly without scrolling pager. |
| `--no-replace-objects` | Ignore replacement objects | `git --no-replace-objects log` | Useful in advanced repos with replacement objects. |
| `--no-lazy-fetch` | Disable lazy fetching | `git --no-lazy-fetch fetch origin` | Forces Git to fetch all objects immediately. |
| `--no-optional-locks` | Skip optional locks | `git --no-optional-locks status` | Avoids optional locks, useful in automation/scripts. |
| `--no-advice` | Suppress advice messages | `git --no-advice status` | Hides hints like “hint: to see more...” |
| `--bare` | Treat repo as bare | `git --bare init myrepo.git` | Creates a repository without a working tree (common for central repos). |
| `--git-dir=<path>` | Specify `.git` directory | `git --git-dir=/srv/git/myrepo.git status` | Use a Git repo stored in a non-standard location. |
| `--work-tree=<path>` | Specify working tree | `git --git-dir=/srv/git/myrepo.git --work-tree=/var/www/html status` | Separates the `.git` folder from the working files. |
| `--namespace=<name>` | Use repository namespace | `git --namespace=featureX log` | Isolates refs under a namespace, advanced multi-repo workflows. |
| `--config-env=<name>=<envvar>` | Read config from environment variable | `export MYEMAIL="me@example.com"` `git --config-env user.email=MYEMAIL commit -m "Env email"` | Uses an environment variable as a config value. |

# Main Porcelain Commands

### **1\. `add`**

**Purpose:** Stage files (add changes to the index) for the next commit.

**Example:**

```
git add file.txt
git add .
```

- `file.txt` → stages a single file
    
- `.` → stages all changes in the current directory
    

* * *

### **2\. `am`**

**Purpose:** Apply patches from emails or mailbox files.

**Example:**

```
git am patchfile.mbox
```

- Applies changes from `patchfile.mbox` to your current branch.

* * *

### **3\. `archive`**

**Purpose:** Create a ZIP or TAR archive of a repository at a specific commit.

**Example:**

```
git archive -o latest.zip HEAD
```

- Creates `latest.zip` containing the files from the latest commit (`HEAD`).

* * *

### **4\. `backfill`**

**Purpose:** Download missing objects in a partial clone.

**Example:**

```
git backfill
```

- Only used in **partial clones** to fetch missing objects needed for operations.

* * *

### **5\. `bisect`**

**Purpose:** Use binary search to find the commit that introduced a bug.

**Example:**

```
git bisect start
git bisect bad # current commit is bad
git bisect good v1.0 # known good commit
```

- Git will guide you through commits to test until it finds the first bad commit.

* * *

### **6\. `branch`**

**Purpose:** List, create, or delete branches.

**Example:**

```
git branch # list branches
git branch feature-x # create new branch
git branch -d feature-x # delete branch
```

* * *

### **7\. `bundle`**

**Purpose:** Create a single file containing objects and refs to share a repository.

**Example:**

```
git bundle create repo.bundle main
```

- Packs the `main` branch into `repo.bundle` for transport.

* * *

### **8\. `checkout`**

**Purpose:** Switch branches or restore files.

**Example:**

```
git checkout feature-x # switch to branch
git checkout -- file.txt # restore file to last committed state
```

* * *

### **9\. `cherry-pick`**

**Purpose:** Apply changes from a specific commit onto your current branch.

**Example:**

```
git cherry-pick 1a2b3c4
```

- Applies commit `1a2b3c4` to your branch without merging the whole branch.

* * *

### **10\. `citool`**

**Purpose:** Graphical commit tool (GUI for `git commit`).

**Example:**

```
git citool
```

- Opens a GUI to select files and write commit messages.

* * *

### **11\. `clean`**

**Purpose:** Remove untracked files from your working directory.

**Example:**

```
git clean -f # remove untracked files
git clean -fd # remove untracked files and directories
```

* * *

### **12\. `clone`**

**Purpose:** Clone a repository into a new directory.

**Example:**

```
git clone https://github.com/user/repo.git
```

- Creates a local copy of the remote repository.

* * *

### **13\. `commit`**

**Purpose:** Record staged changes in the repository.

**Example:**

```
git commit -m "Add new feature"
```

- Commits all staged changes with the given message.

* * *

### **14\. `describe`**

**Purpose:** Describe a commit in human-readable form using nearest tag.

**Example:**

```
git describe
```

- Outputs something like `v1.0-5-gd3f2a1` (5 commits past tag `v1.0`).

* * *

### **15\. `diff`**

**Purpose:** Show changes between commits, branches, or working directory.

**Example:**

```
git diff # unstaged changes
git diff --staged # staged changes
git diff HEAD~1 HEAD # changes between last 2 commits
```

* * *

### **16\. `fetch`**

**Purpose:** Download objects and refs from a remote repository without merging.

**Example:**

```
git fetch origin
```

- Fetches all changes from `origin` but doesn’t merge them into current branch.

* * *

### **17\. `format-patch`**

**Purpose:** Prepare commits as patch files for email submission.

**Example:**

```
git format-patch -3
```

- Creates patch files for the last 3 commits.

* * *

### **18\. `gc`**

**Purpose:** Cleanup and optimize the repository.

**Example:**

```
git gc
```

- Compresses unnecessary files and optimizes repo performance.

* * *

### **19\. `gitk`**

**Purpose:** Graphical repository browser.

**Example:**

```
gitk
```

- Opens a GUI to visualize commit history.

* * *

### **20\. `grep`**

**Purpose:** Search for text in the repository.

**Example:**

```
git grep "functionName"
```

- Lists all lines containing `functionName`.

* * *

### **21\. `gui`**

**Purpose:** Git portable graphical interface.

**Example:**

```
git gui
```

- Opens a GUI for staging, committing, and browsing changes.

* * *

### **22\. `init`**

**Purpose:** Initialize a new Git repository.

**Example:**

```
git init my-project
```

- Creates a `.git` directory and starts tracking files.

* * *

### **23\. `log`**

**Purpose:** Show commit history.

**Example:**

```
git loggit log --oneline
git log --graph --all
```

- Displays commits, optionally condensed or as a graph.

* * *

### **24\. `maintenance`**

**Purpose:** Run tasks to optimize Git repository.

**Example:**

```
git maintenance start
git maintenance stop
```

- Useful for automatic maintenance on large repos.

* * *

### **25\. `merge`**

**Purpose:** Merge another branch into your current branch.

**Example:**

```
git merge feature-x
```

- Brings changes from `feature-x` into the current branch.

* * *

### **26\. `mv`**

**Purpose:** Rename or move files in Git.

**Example:**

```
git mv old.txt new.txt
```

- Stages the rename automatically.

* * *

### **27\. `notes`**

**Purpose:** Add or view notes attached to commits.

**Example:**

```
git notes add -m "Reviewed by QA"
git log --show-notes
```

* * *

### **28\. `pull`**

**Purpose:** Fetch and merge changes from a remote.

**Example:**

```
git pull origin main
```

- Fetches from `origin` and merges into the current branch.

* * *

### **29\. `push`**

**Purpose:** Upload local commits to a remote repository.

**Example:**

```
git push origin main
```

- Updates the `main` branch on the remote repository.

* * *

### **30\. `range-diff`**

**Purpose:** Compare two commit ranges.

**Example:**

```
git range-diff main..feature-x main..feature-y
```

- Compares how the commits differ between two versions of a branch.

* * *

### **31\. `rebase`**

**Purpose:** Move or reapply commits on top of another branch.

**Example:**

```
git rebase main
```

- Reapplies your current branch commits on top of `main`.

* * *

### **32\. `reset`**

**Purpose:** Reset current branch to a specific state.

**Example:**

```
git reset --soft HEAD~1 # keep changes staged
git reset --hard HEAD~1 # discard changes
```

* * *

### **33\. `restore`**

**Purpose:** Restore files in working directory.

**Example:**

```
git restore file.txt
git restore --staged file.txt
```

- Restores last committed version (either from working tree or staging area).

* * *

### **34\. `revert`**

**Purpose:** Undo a commit by creating a new commit.

**Example:**

```
git revert 1a2b3c
```

- Reverts commit `1a2b3c` without rewriting history.

* * *

### **35\. `rm`**

**Purpose:** Remove files from working tree and staging.

**Example:**

```
git rm file.txt
```

- Stages file removal.

* * *

### **36\. `scalar`**

**Purpose:** Tool to manage very large repositories efficiently.

**Example:**

```
git scalar clone https://github.com/large/repo.git
```

* * *

### **37\. `shortlog`**

**Purpose:** Summarize git log output.

**Example:**

```
git shortloggit shortlog -s -n
```

- Shows commits per author (`-s`) sorted by number (`-n`).

* * *

### **38\. `show`**

**Purpose:** Show info about objects, commits, files.

**Example:**

```
git show HEAD
git show 1a2b3c
```

* * *

### **39\. `sparse-checkout`**

**Purpose:** Work with only a subset of files in a repo.

**Example:**

```
git sparse-checkout init --cone
git sparse-checkout set src/
```

* * *

### **40\. `stash`**

**Purpose:** Temporarily save uncommitted changes.

**Example:**

```
git stashgit stash pop
```

* * *

### **41\. `status`**

**Purpose:** Show the current state of working directory and staging area.

**Example:**

```
git status
```

### **42\. `submodule`**

**Purpose:** Manage submodules (nested Git repos).

**Example:**

* * *

### **43\. `switch`**

**Purpose:** Switch branches (simpler alternative to `checkout`).

**Example:**

* * *

### **44\. `tag`**

### 1\. **`git config`** – Get and set repository or global options

This command is used to configure Git behavior, either **globally** (for all repositories) or **locally** (per repository).

**Example:**

```#
git config --global user.name "Sushil Choudhary"

# Set the email for commits in this repo only
git config user.email "sushil@example.com"

# Check your configuration
git config --list
```

* * *

### 2\. **`git fast-export`** – Git data exporter

Exports Git repository data in a stream format suitable for importing elsewhere. Often used for **migrating repositories**.

**Example:**

```
# Export all branches to a file
git fast-export --all > repo.export

# You can import this file into another repository using git fast-import
```

* * *

### 3\. **`git fast-import`** – Backend for fast Git data importers

Imports data produced by `git fast-export`. This is mostly **used in automated scripts or large repository migrations**.

**Example:**

```
# Create a new repository
mkdir newrepo && cd newrepo
git init

# Import previously exported data
git fast-import < ../repo.export
```

* * *

### 4\. **`git filter-branch`** – Rewrite branches

Rewrites Git history, e.g., **removing a file from all commits** or **changing author info**. This is **destructive**: it rewrites commits.

**Example:**

```
# Remove a large file from all commits
git filter-branch --tree-filter 'rm -f secret.txt' -- --all

# Change the author name for all commits
git filter-branch --env-filter '
export GIT_AUTHOR_NAME="New Name"
export GIT_COMMITTER_NAME="New Name"
' -- --all
```

> Warning: Use carefully; this rewrites history and may break collaboration if pushed to a shared repo.

* * *

### 5\. **`git mergetool`** – Run merge conflict resolution tools

Helps you resolve merge conflicts using an external GUI/text tool.

**Example:**

```
# Run your configured merge tool after a conflict
git mergetool

# You can configure a merge tool, e.g., vimdiff
git config --global merge.tool vimdiff
```

* * *

### 6\. **`git pack-refs`** – Pack heads and tags

Stores references (branches and tags) in a **single file** for faster access. Useful in **large repositories**.

**Example:**

```
# Pack refs for efficiency
git pack-refs
```

* * *

### 7\. **`git prune`** – Prune unreachable objects

Removes **objects not reachable from any branch or tag**. Helps **clean up repository size**.

**Example:**

```
# Remove unreachable objects
git prune

# Often used after garbage collection
git gc
```

* * *

### 8\. **`git reflog`** – Manage reflog information

Shows a **history of HEAD changes**, even if commits are no longer in a branch.

**Example:**

```
# Show the last 10 HEAD movements
git reflog -10

# Recover a deleted commit
git checkout HEAD@{3}
```

* * *

### 9\. **`git refs`** – Low-level access to refs

Manipulate or inspect references (branches, tags, remotes) at a low level.

**Example:**

```
# Show all refs
git show-ref
```

* * *

### 10\. **`git remote`** – Manage set of tracked repositories

Adds, removes, or lists **remote repositories**.

**Example:**

```
# List remotes
git remote -v

# Add a remote
git remote add origin https://github.com/user/repo.git

# Remove a remote
git remote remove origin
```

* * *

### 11\. **`git repack`** – Pack unpacked objects

Packs loose objects into **packfiles** for better storage and performance.

**Example:**

```
# Repack the repository
git repack -a -d -l
```

* * *

### 12\. **`git replace`** – Replace objects

Allows one object (commit/blob/tree) to temporarily **replace another**. Useful for testing or fixing a bad commit without rewriting history.

**Example:**

```
# Replace commit abc123 with def456
git replace abc123 def456

# List replacements
git replace -l

# Delete a replacement
git replace -d abc123
```

# Ancillary Commands / Interrogators

These **interrogator commands** are mainly used to **inspect, analyze, verify, and debug** Git repositories. They do not usually change history or data; instead, they help you understand *what exists*, *what changed*, and *why*.

* * *

## 1\. `git annotate`

**Purpose:** Annotate each line of a file with the commit that last modified it.

> `git annotate` is an older name; today, `git blame` is preferred.

### Example

```bash
git annotate README.md
```

### What you see

- Commit hash
    
- Author
    
- Line number
    
- Content of the line
    

**Use case:** Finding when a line was introduced and by whom.

* * *

## 2\. `git blame`

**Purpose:** Show which commit and author last modified each line of a file.

### Example

```bash
git blame app.js
```

### Practical usage

```bash
git blame -L 10,20 app.js
```

Shows blame information only for lines 10–20.

**Use case:** Debugging bugs or understanding code ownership.

* * *

## 3\. `git bugreport`

**Purpose:** Collect system and Git environment information to help file a Git bug report.

### Example

```bash
git bugreport
```

### Output

- Creates a `.zip` file
    
- Includes Git version, config, OS details
    

**Use case:** When reporting Git bugs to Git maintainers.

* * *

## 4\. `git count-objects`

**Purpose:** Count loose objects and report disk usage.

### Example

```bash
git count-objects -v
```

### Sample output

- Number of loose objects
    
- Size on disk
    
- Packfile statistics
    

**Use case:** Checking repository size and health.

* * *

## 5\. `git diagnose`

**Purpose:** Generate a ZIP archive with detailed diagnostic data.

### Example

```bash
git diagnose
```

### What it collects

- Config files
    
- Refs
    
- Object statistics
    

**Use case:** Advanced troubleshooting or reporting performance issues.

* * *

## 6\. `git difftool`

**Purpose:** Show differences using an external diff tool.

### Example

```bash
git difftool
```

### Configure a diff tool

**Use case:** Visual comparison of changes instead of terminal diff.

## 7\. `git fsck`

**Purpose:** Verify integrity and connectivity of Git objects.

### Example

```bash
git fsck
```

### Detects

- Missing objects
    
- Corrupt objects
    
- Broken links
    

**Use case:** Recovering from disk corruption or crashes.

### Example

**Use case:** Hosting a lightweight web view of repositories (mostly server-side).

# Git Low-level Commands / Syncing Repositories

These commands form the **transport and synchronization layer of Git**. They are responsible for **moving objects and refs between repositories**. Porcelain commands like `git fetch`, `git pull`, and `git push` are built on top of these plumbing commands.

You rarely use these directly unless you are:

- Implementing Git servers or tooling
    
- Debugging fetch/push problems
    
- Learning Git internals deeply
    

* * *

## 1\. `git daemon`

**Purpose:** Run a very simple Git server that serves repositories over the `git://` protocol.

### Example: Start a Git daemon

```bash
git daemon --reuseaddr --base-path=/srv/git /srv/git
```

### What it does

- Exposes repositories as read-only by default
    
- Uses the unauthenticated `git://` protocol
    

**Use case:** Internal networks, mirrors, or lightweight read-only servers.

* * *

## 2\. `git fetch-pack`

**Purpose:** Fetch missing objects and refs from another repository.

> This is the **engine behind `git fetch`**.

### Example

```bash
git fetch-pack https://example.com/repo.git refs/heads/main
```

### What it does

- Negotiates object transfer
    
- Downloads only missing objects
    

**Use case:** Custom fetch implementations or debugging fetch issues.

* * *

## 3\. `git http-backend`

**Purpose:** Server-side implementation of Git over HTTP/HTTPS.

### Example (used with a web server)

```bash
git http-backend
```

### How it’s used

- Invoked by Apache, Nginx, or CGI
    
- Handles smart HTTP Git requests
    

**Use case:** Hosting Git repositories over HTTP(S).

* * *

## 4\. `git send-pack`

**Purpose:** Send objects and refs to another repository.

> This is the **engine behind `git push`**.

### Example

```bash
git send-pack origin refs/heads/main
```

### What it does

- Transfers commits, trees, and blobs
    
- Updates remote refs
    

**Use case:** Low-level push operations or debugging push failures.

* * *

## 5\. `git update-server-info`

**Purpose:** Generate auxiliary files for **dumb servers** (non-smart HTTP or FTP).

### Example

```bash
git update-server-info
```

### What it updates

- `info/refs`
    
- `objects/info/packs`
    

**Use case:** Making repositories accessible via static file servers.

* * *

## How These Fit Together (Important)

### Fetch flow

```text
fetch → fetch-pack → receive objects → update refs
```

### Push flow

```text
push → send-pack → transfer objects → update remote refs
```

* * *

## Porcelain vs Plumbing Mapping

| Porcelain Command | Plumbing Used |
| --- | --- |
| git fetch | fetch-pack |
| git pull | fetch-pack + merge/rebase |
| git push | send-pack |
| git clone | fetch-pack + checkout |

* * *

## Safety Notes

- These commands **can modify remote state**
    
- Incorrect usage may corrupt repositories
    
- Always test in disposable repositories

# Git User-facing Repository, Command, and File Interfaces

This document explains **Git’s user-facing interfaces**—configuration files, conventions, and concepts that developers interact with directly. These define **how Git behaves**, **how commands are interpreted**, and **how repositories are structured**.

Unlike plumbing commands, these are **essential for everyday Git usage**.

* * *

## 1\. Git Attributes (`.gitattributes`)

**Purpose:** Define attributes for paths that affect how Git handles files.

### Common uses

- Line ending normalization
    
- Custom diff or merge drivers
    
- Binary file handling
    

### Example: `.gitattributes`

```gitattributes
*.txt text
*.png binary
*.c diff=cpp
```

### Inspect attributes

```bash
git check-attr text -- file.txt
```

**Use case:** Enforcing consistent behavior across platforms and teams.

* * *

## 2\. Git CLI (`git` command-line interface)

**Purpose:** Defines how Git commands, options, and conventions work.

### Command structure

```text
git <command> [options] [arguments]
```

### Examples

```bash
git status
git commit -m "Initial commit"
git log --oneline --graph
```

### Conventions

- Short options: `-m`
    
- Long options: `--amend`
    
- Global options before command
    

**Use case:** Understanding Git’s consistent command patterns.

* * *

## 3\. Git Hooks (`.git/hooks/`)

**Purpose:** Scripts that run automatically at specific points in Git workflows.

### Common hooks

- `pre-commit`
    
- `commit-msg`
    
- `pre-push`
    

### Example: `pre-commit` hook

```bash
#!/bin/sh
git diff --cached --check || exit 1
```

Make executable:

```bash
chmod +x .git/hooks/pre-commit
```

**Use case:** Enforcing code quality, tests, or policies.

* * *

## 4\. Git Ignore (`.gitignore`)

**Purpose:** Specify intentionally untracked files Git should ignore.

### Example: `.gitignore`

```gitignore
node_modules/
*.log
.env
```

### Debug ignored files

```bash
git check-ignore -v debug.log
```

**Use case:** Prevent committing build artifacts or secrets.

* * *

## 5\. Git Mailmap (`.mailmap`)

**Purpose:** Map multiple author identities to a canonical name/email.

### Example: `.mailmap`

```text
Sushil Chaudhary <sushil@example.com> <old@mail.com>
```

### Apply mailmap

```bash
git log --use-mailmap
```

**Use case:** Clean contributor history and statistics.

* * *

## 6\. Git Modules (`.gitmodules`)

**Purpose:** Define submodule properties for repositories embedded inside another.

### Example: `.gitmodules`

```ini
[submodule "libmath"]
  path = libmath
  url = https://github.com/example/libmath.git
```

### Initialize submodules

```bash
git submodule init

```

**Use case:** Reusing external repositories as dependencies.

## 7\. Git Repository Layout

**Purpose:** Defines the internal structure of a Git repository.

### Common layout

```text
.git/
├── HEAD
├── config
├── objects/
├── refs/
│   ├── heads/
│   └── tags/
├── index
└── hooks/
```

### Inspect repository

```bash
ls .git
```

**Use case:** Debugging, scripting, and understanding Git internals.

* * *

## 8\. Git Revisions (`gitrevisions`)

**Purpose:** Specify commits, branches, tags, and commit ranges.

### Single revisions

```bash
git show HEAD
git show main~1
git show v1.0.0
```

### Ranges

```bash
git log main..feature
git log HEAD~5..HEAD
```

### Special syntax

```text
HEAD^   # first parent
HEAD~2  # two commits back
```

**Use case:** Precise history inspection and scripting.

# Git Developer-facing File Formats, Protocols, and Interfaces

This document covers **developer-facing Git internals**: file formats and network protocols that Git uses to **store data efficiently** and **communicate between repositories**. These are primarily relevant if you are:

- Building Git tooling or integrations
    
- Working on Git itself
    
- Debugging low-level performance or corruption issues
    
- Learning Git internals deeply
    

These are **not everyday user concepts**, but they explain *why Git works the way it does*.

* * *

## 1\. `format-bundle` – Git Bundle File Format

**Purpose:** Describe the `.bundle` file format, which packages Git objects and refs into a single portable file.

### What a bundle is

- A self-contained Git repository snapshot
    
- Can be transferred without network access
    

### Example: Create a bundle

```bash
git bundle create repo.bundle main
```

### Use a bundle

```bash
git clone repo.bundle myrepo
```

**Use case:** Offline transfers, backups, air-gapped environments.

* * *

## 2\. `format-chunk` – Chunk-based File Formats

**Purpose:** Define a common chunked file layout used by several Git data files.

### Key ideas

- File = header + chunks + footer
    
- Each chunk has a type and size
    

### Used by

- Commit-graph files
    
- Multi-pack-index
    

**Use case:** Allowing Git to extend formats without breaking compatibility.

* * *

## 3\. `format-commit-graph` – Commit-Graph Format

**Purpose:** Describe how Git stores commit graph data for fast history traversal.

### Why it exists

- Avoids repeatedly parsing commit objects
    
- Speeds up `log`, `merge-base`, `rev-list`
    

### Example: Generate commit-graph

```bash
git commit-graph write --reachable
```

**Use case:** Large repositories with deep history.

* * *

## 4\. `format-index` – Git Index Format

**Purpose:** Define the binary format of the Git index (`.git/index`).

### What the index stores

- File paths
    
- Blob hashes
    
- File modes
    
- Staging information
    

### Related command

```bash
git ls-files -s
```

**Use case:** Understanding staging behavior and performance.

* * *

## 5\. `format-pack` – Git Pack File Format

**Purpose:** Describe how Git stores many objects efficiently in `.pack` files.

### Key features

- Delta compression
    
- Binary efficient storage
    

### Example: Create pack files

```bash
git gc
git repack
```

**Use case:** Reducing disk usage and speeding up transfers.

* * *

## 6\. `format-signature` – Cryptographic Signature Formats

**Purpose:** Describe formats for signed commits and tags.

### Example: Signed commit

```bash
git commit -S -m "Signed commit"
```

### Verify signature

```bash
git verify-commit HEAD
```

**Use case:** Authenticity and supply-chain security.

* * *

## 7\. `protocol-capabilities` – Protocol v0 and v1 Capabilities

**Purpose:** Define feature negotiation between Git clients and servers.

### Examples of capabilities

- `multi_ack`
    
- `thin-pack`
    
- `side-band`
    

### Concept

- Client and server agree on supported features before transfer

**Use case:** Backward compatibility and optimization.

* * *

## 8\. `protocol-common` – Common Protocol Concepts

**Purpose:** Describe shared ideas across Git protocols.

### Includes

- pkt-line format
    
- Object negotiation
    
- Error handling
    

### Example: pkt-line

```text
0032want <object-id>\n
```

**Use case:** Implementing Git-compatible clients or servers.

* * *

## 9\. `protocol-http` – Git over HTTP

**Purpose:** Define how Git communicates over HTTP/HTTPS.

### Modes

- Dumb HTTP (static files)
    
- Smart HTTP (RPC-based)
    

### Example

```text
GET /repo.git/info/refs?service=git-upload-pack
```

**Use case:** Hosting Git repositories behind firewalls.

* * *

## 10\. `protocol-pack` – Pack Transfer Protocol

**Purpose:** Describe how pack files are transferred over the wire.

### Transfer flow

```text
negotiate objects → send pack → update refs
```

### Used by

- fetch
    
- push
    

**Use case:** Efficient network transfer of repository data.

* * *

## 11\. `protocol-v2` – Git Wire Protocol Version 2

**Purpose:** Modern, extensible Git protocol.

### Advantages

- Clear command separation
    
- Improved performance
    
- Easier extensibility
    

### Example concept

```text
command=fetch
command=ls-refs
```

**Use case:** Modern Git servers and clients.

* * *

## Big Picture Overview

```text
Objects → Pack files → Protocol → Remote repository
```

Git is fundamentally a **content-addressed database** with **efficient synchronization protocols**.
